#  HOW TO RUN
#     ./sim -schd priority tests/input_sched_priority.txt
#     ./sim -schd sjf      tests/input_sched_priority.txt   (to compare)
#     ./sim -schd fcfs     tests/input_sched_priority.txt   (to compare)
#
#  DESIGN
#     7 travelers start on 7 DIFFERENT nodes (1..7), each exactly 1 weight
#     unit from the central collision node 0. All 7 reach node 0 at the
#     same instant and must queue. Only one traveler fits in a node, so six
#     BLOCK and the parent drains the queue in the algorithm's order.
#
#  PRIORITY RULE IN THE CODE:  sched_priority[i] = K - i   (K = 7)
#        T0 -> priority 7  (HIGHEST)   ... T6 -> priority 1 (LOWEST)
#     Scheduler admits the HIGHEST priority waiting traveler first.
#
#  PATH COSTS (node 0's out-edges are weighted to DECREASE with index, so
#  SJF order is the REVERSE of PRIORITY order — that's the key contrast):
#        T0 : 1->0->8    total cost 14   (priority 7, most EXPENSIVE path)
#        T1 : 2->0->9    total cost 12
#        T2 : 3->0->10   total cost 10
#        T3 : 4->0->11   total cost  8
#        T4 : 5->0->12   total cost  6
#        T5 : 6->0->13   total cost  4
#        T6 : 7->0->14   total cost  2   (priority 1, CHEAPEST path)
#
#  EXPECTED ADMISSION ORDER AT NODE 0  (read the "entered node 0" lines):
#        PRIORITY ->  T0, T1, T2, T3, T4, T5, T6   (highest priority first)
#        SJF      ->  T6, T5, T4, T3, T2, T1, T0   (cheapest path first = REVERSE)
#        FCFS     ->  arrival order (roughly T0..T6, whoever's write lands first)
#
#  THE PROOF: PRIORITY and SJF produce OPPOSITE orders on the same graph.
#  If your priority run shows T0 entering node 0 first (and SJF shows T6
#  first), the priority policy is working correctly.
# ════════════════════════════════════════════════════════════════════
